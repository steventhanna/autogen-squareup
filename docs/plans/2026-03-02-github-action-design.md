# GitHub Action: Auto-Update from Square Spec ŌĆö Design Document

## Problem

The Square API spec updates roughly monthly. We need an automated pipeline that detects new spec versions, regenerates the crate, and opens a PR for review.

## Solution

Two GitHub Action workflows:

1. **Check for updates** (scheduled) ŌĆö detects new Square spec versions, regenerates, opens a PR
2. **Tag on merge** (triggered on PR merge) ŌĆö tags the merge commit with the crate version

## Version Tracking

Square uses date-based API versioning (`YYYY-MM-DD`), embedded in the spec as the `Square-Version` header default value.

Our crate version encodes this directly: `0.YYYYMMDD.patch`
- `2025-10-16` ŌåÆ `0.20251016.0`
- Internal bugfix ŌåÆ `0.20251016.1`
- `2026-01-22` ŌåÆ `0.20260122.0`

### Extracting the Square Version from the Spec

```bash
jq -r '.components.securitySchemes.oauth2."x-additional-headers"[0].schema.default' api.json
# Returns: "2025-10-16"
```

### Converting to Crate Version

```bash
# "2025-10-16" ŌåÆ "0.20251016.0"
SQUARE_DATE="2025-10-16"
CRATE_VERSION="0.$(echo $SQUARE_DATE | tr -d '-').0"
```

## Workflow 1: Check for Updates

**Trigger:** Cron (Monday + Thursday 9am UTC) + manual dispatch

```yaml
on:
  schedule:
    - cron: '0 9 * * 1,4'
  workflow_dispatch:
```

**Steps:**

1. Checkout repo
2. Setup Java (for openapi-generator)
3. Setup Rust toolchain
4. Install openapi-generator
5. Fetch latest spec from Square master
6. Extract Square-Version date from spec
7. Compare against current Cargo.toml version
8. If same: exit early (no update needed)
9. If different:
   a. Run `./generate.sh` (regenerate + post-process + cargo check)
   b. Update `Cargo.toml` version to `0.YYYYMMDD.0`
   c. Update `generate.sh` packageVersion to match
   d. Run `cargo test --all-features`
   e. Check if a PR already exists for this version (prevent duplicates)
   f. Commit all changes to branch `update/square-api-YYYY-MM-DD`
   g. Push branch and create PR

**PR format:**

```
Title: Update to Square API YYYY-MM-DD

Body:
## Square API Update

- **Old version:** 2025-10-16 (crate 0.20251016.0)
- **New version:** 2026-01-22 (crate 0.20260122.0)
- **cargo check:** Ō£ģ passed
- **cargo test:** Ō£ģ passed (N tests)

Generated by the automated spec update workflow.
```

## Workflow 2: Tag on Merge

**Trigger:** PR merge to main where branch matches `update/square-api-*`

```yaml
on:
  pull_request:
    types: [closed]
    branches: [main]
```

**Steps:**

1. Check if PR was merged (not just closed)
2. Check if branch name matches `update/square-api-*`
3. Extract version from Cargo.toml
4. Create and push git tag `v{version}` (e.g., `v0.20260122.0`)

## Dependencies

The update workflow needs:
- `actions/checkout`
- `actions/setup-java` (for openapi-generator CLI)
- `dtolnay/rust-toolchain@stable`
- `openapi-generator` CLI (installed via npm or direct download)
- `peter-evans/create-pull-request` or `gh` CLI for PR creation

## Duplicate Prevention

Before creating a PR, check:
```bash
gh pr list --head "update/square-api-$SQUARE_DATE" --state open
```
If a PR already exists for this version, skip.

## Decisions Log

| Decision | Choice | Reason |
|----------|--------|--------|
| Crate versioning | `0.YYYYMMDD.patch` | Encodes Square date directly, auto-incrementable |
| Schedule | Twice weekly (Mon + Thu) | Max 3-4 day lag, ~8 runs/month |
| Merge strategy | Auto-PR for review | Safe ŌĆö review generated changes before merging |
| Past versions | Git tags per release | Standard approach, crates.io retains all published versions |
| Tagging | On PR merge | Only approved versions get tagged |
